# Hermes iOS 1.1 Production Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:test-driven-development` for behavior changes and `superpowers:verification-before-completion` before marking any phase complete. Keep this document updated as work progresses.

**Goal:** Turn Hermes iOS from a proven companion-app concept into a production-ready Hermes client with clear managed, Tailscale, and self-hosted relay paths.

**Architecture:** Hermes keeps its relay/connector split: iOS talks to a Hermes relay, the relay talks to the connector over WebSocket, and the connector executes Hermes locally. Production push is handled through an official managed push broker with App Attest and relay-scoped push grants, so self-hosted relays never receive official APNs credentials or raw APNs tokens.

**Tech Stack:** SwiftUI + Observation, StoreKit 2 later, DeviceCheck/App Attest, UserNotifications/APNs, FastAPI, SQLAlchemy, SQLite WAL on Fly volumes initially, PostgreSQL later if multi-node or admin requirements demand it.

## Product Modes

### Managed Relay

The managed relay is the default best-experience path. It should provide public reachability, job queueing while the host reconnects, official APNs, hosted push broker registration, and later StoreKit-gated access.

### Self-Hosted Tailscale

Tailscale is the private network path. It should use the existing Hermes relay API, but the relay runs on the user's Mac or trusted host and is reached over a tailnet URL or tailnet IP. It should not promise official APNs. It should be honest about foreground/reconnect behavior and guide the user to open Tailscale when unreachable.

### Self-Hosted Relay URL

The custom relay URL path is for users who operate their own reachable Hermes relay. It should support full relay behavior when the user-owned relay is reachable. Official APNs should require managed push grants, not user access to Hermes' APNs credentials.

## Where Hermes Differs From OpenClaw

OpenClaw's iOS app talks directly to a gateway WebSocket. Hermes should not copy that. Hermes' production shape should preserve:

- iOS -> relay HTTP/SSE.
- Relay -> connector WebSocket jobs/RPC.
- Connector-owned Hermes execution.
- Connector-local sensor SQLite.
- Connector-local OpenAI Realtime API key.
- Realtime audio/camera direct to OpenAI/WebRTC, not through the Hermes relay.

The OpenClaw reference is useful for product boundaries:

- QR-first setup with manual fallback.
- Home/Tailscale vs remote-domain mode clarity.
- Read-only or fail-fast behavior when a private gateway is unreachable.
- Official-build push relay instead of sharing APNs credentials with self-hosters.

## Official References

- Apple App Attest: `DCAppAttestService` validates legitimate app instances and supports key generation, attestation, and assertions.
- Apple UserNotifications: background pushes can wake the app, but delivery is not guaranteed and excessive background pushes can be throttled.
- Apple BackgroundTasks: background work should be scheduled and bounded; sockets should not be treated as continuously alive in background.
- Tailscale Serve: tailnet-only service exposure can proxy localhost services and provides identity headers for Serve traffic.
- Fly Volumes: volumes are single-region, one-machine-attached persistent disks and are not automatically replicated.
- SQLite WAL: WAL improves read/write concurrency, but SQLite still has one writer at a time and needs checkpoint awareness.

## 2026 Validation Notes

- App Attest requires a unique one-time server challenge before attestation or assertion. The server must store the challenge, reject replay, validate CBOR attestation/assertion objects, bind the RP ID hash to the app ID, store the attested public key/receipt, and verify assertion counters for later protected requests. Reference: <https://developer.apple.com/documentation/devicecheck/validating-apps-that-connect-to-your-server>
- StoreKit 2 transaction and app-transaction data are JWS-backed and can be verified on device or server; production managed relay entitlement should eventually use StoreKit 2 current entitlements plus server-side JWS validation, not local flags. References: <https://developer.apple.com/documentation/storekit/transaction>, <https://developer.apple.com/documentation/storekit/apptransaction>
- APNs provider servers use HTTP/2/TLS to Apple endpoints. Background pushes are useful for wake/reconnect but cannot be treated as guaranteed execution. References: <https://developer.apple.com/documentation/usernotifications/setting-up-a-remote-notification-server>, <https://developer.apple.com/documentation/usernotifications/pushing-background-updates-to-your-app>
- Fly Volumes are persistent but single-machine/single-region and are not automatically replicated. The SQLite managed-beta path must remain single-node with backups until LiteFS/Postgres or another replication layer is adopted. Reference: <https://fly.io/docs/volumes/overview/>
- Tailscale Serve is a private reachability option, not an APNs solution. It can proxy local services into a tailnet, but official APNs still needs Hermes-managed broker infrastructure. Reference: <https://tailscale.com/docs/features/tailscale-serve>
- OpenClaw's iOS app is a reference for product boundaries: official builds use a push relay with App Attest/app receipt; local/self-hosted builds do not receive official APNs credentials. Hermes should copy that trust split, not OpenClaw's direct-gateway WebSocket architecture. Reference: <https://github.com/openclaw/openclaw/tree/main/apps/ios>
